## Purpose
Ensure new team members in any role quickly understand their responsibilities, key relationships, and how to be successful in the OctoAcme project environment.

## How to Use This Template
- Project Manager or Scrum Master should use this checklist for each new team member
- Customize based on the specific role and project context
- Track completion in an issue or project board
- Schedule a follow-up at 30 days to gather feedback and address gaps

---

## General Onboarding (All Roles)

**Day 1-2: Foundation**
- [ ] Review [OctoAcme Project Management Overview](octoacme-project-management-overview.md)
- [ ] Review [Roles and Personas](octoacme-roles-and-personas.md) to understand your role and team structure
- [ ] Understand the project lifecycle phases
- [ ] Access granted to:
  - [ ] GitHub repository
  - [ ] Project board
  - [ ] Communication channels (Slack, Teams, etc.)
  - [ ] Documentation tools (Figma, Confluence, etc.)
- [ ] Introduction to the team in standup or team meeting
- [ ] Assigned a buddy/mentor for the first 2 weeks

**Week 1: Context and Relationships**
- [ ] Review current project one-pager and success metrics
- [ ] Review recent sprint goals and retrospective notes
- [ ] Shadow key meetings: standup, sprint planning, or design review
- [ ] Meet 1:1 with Project Manager to understand current priorities and risks
- [ ] Meet 1:1 with Product Manager to understand product vision and roadmap
- [ ] Understand blocker escalation process

**Week 2: Active Contribution**
- [ ] Assigned first task or story with clear acceptance criteria
- [ ] Paired with a teammate on first deliverable
- [ ] First contribution reviewed and merged
- [ ] Familiar with PR workflow and quality standards
- [ ] Attended first retrospective (if applicable)

## Role-Specific Onboarding

### For Developers

**Technical Setup**
- [ ] Development environment configured and tested
- [ ] CI/CD pipeline access and understanding
- [ ] Code review guidelines reviewed
- [ ] Branching and PR conventions understood
- [ ] Linting and testing tools running locally

**Knowledge Transfer**
- [ ] Codebase architecture overview completed
- [ ] Key services and dependencies mapped
- [ ] Testing strategy explained (unit, integration, E2E)
- [ ] Deployment process reviewed with DevOps Engineer

**First Contributions**
- [ ] Complete a "good first issue" or bug fix
- [ ] Participate in code review (give and receive feedback)
- [ ] Attend technical design discussion

---

### For UX Designers

**Design Setup**
- [ ] Access to design tools (Figma, Sketch, etc.) and component libraries
- [ ] Understand design handoff process to Developers
- [ ] Accessibility standards (WCAG) documentation reviewed
- [ ] User research repository and insights shared

**Knowledge Transfer**
- [ ] Review existing design patterns and style guide
- [ ] Understand current user personas and journey maps
- [ ] Walkthrough of recent UX decisions and rationale
- [ ] Introduction to key Product Manager and Developer partners

**First Contributions**
- [ ] Shadow a design review meeting
- [ ] Contribute feedback to an in-progress design
- [ ] Complete a small design task (icon, component update, etc.)
- [ ] Participate in user testing or feedback session

---

### For DevOps Engineers

**Infrastructure Setup**
- [ ] Access to cloud environments (AWS, Azure, GCP, etc.)
- [ ] Access to CI/CD tools (GitHub Actions, Jenkins, etc.)
- [ ] Monitoring and alerting dashboards reviewed
- [ ] Incident response playbook and on-call rotation explained

**Knowledge Transfer**
- [ ] Infrastructure architecture diagram reviewed
- [ ] Deployment pipeline walkthrough
- [ ] Security scanning and compliance requirements
- [ ] Disaster recovery and rollback procedures

**First Contributions**
- [ ] Shadow a deployment or release
- [ ] Review and improve CI/CD pipeline configuration
- [ ] Participate in post-mortem or incident review
- [ ] Update monitoring or alerting for a service

---

### For Product Managers

**Product Context**
- [ ] Product vision, mission, and strategy documents reviewed
- [ ] Customer segments and personas understood
- [ ] Current roadmap and prioritization framework explained
- [ ] Key metrics and success criteria dashboard access

**Stakeholder Mapping**
- [ ] Introduction to executive sponsors and key stakeholders
- [ ] Understand cross-team dependencies
- [ ] Customer feedback channels identified (support tickets, surveys, user research)

**First Contributions**
- [ ] Shadow customer interview or usability test
- [ ] Review and prioritize backlog items
- [ ] Write acceptance criteria for upcoming feature
- [ ] Present roadmap update in stakeholder sync

---

### For Project Managers

**Project Setup**
- [ ] Access to project boards, timelines, and risk registers
- [ ] Understand current project status and upcoming milestones
- [ ] Review stakeholder communication plan and cadence
- [ ] Understand budget and resource allocation

**Process Familiarization**
- [ ] Review [Project Initiation Guide](octoacme-project-initiation.md)
- [ ] Review [Execution & Tracking](octoacme-execution-and-tracking.md) workflows
- [ ] Review [Risk Management & Communication](octoacme-risks-and-communication.md) practices
- [ ] Understand retrospective process and action item tracking

**First Contributions**
- [ ] Facilitate first standup or project sync
- [ ] Update project status report or risk register
- [ ] Resolve a blocker or coordinate a dependency
- [ ] Schedule and run a retrospective

---

### For Scrum Masters

**Agile Process Setup**
- [ ] Access to project boards and sprint planning tools
- [ ] Understand team's current sprint cadence and ceremonies
- [ ] Review team velocity and historical burndown charts
- [ ] Understand Definition of Ready and Definition of Done

**Team Dynamics**
- [ ] Introduction to each team member and their role
- [ ] Review past retrospective themes and action items
- [ ] Understand current team challenges and improvement goals
- [ ] Align with Project Manager on responsibilities and communication

**First Contributions**
- [ ] Co-facilitate a standup or ceremony
- [ ] Run a retrospective independently
- [ ] Coach a team member on estimation or story breakdown
- [ ] Identify and remove a team blocker

---

### For QA/Testing

**Testing Setup**
- [ ] Access to test environments and test data
- [ ] Automated test suite walkthrough
- [ ] Bug tracking and reporting process explained
- [ ] Understand regression testing scope and frequency

**Quality Standards**
- [ ] Review acceptance criteria templates
- [ ] Understand test coverage goals (unit, integration, E2E)
- [ ] Security testing and compliance requirements
- [ ] Performance and load testing requirements (if applicable)

**First Contributions**
- [ ] Execute manual test cases for a feature in QA
- [ ] Log and triage a bug or issue
- [ ] Write or improve an automated test
- [ ] Participate in sprint demo or acceptance review

---

## 30-Day Check-In

**Questions for New Team Member**:
- What has gone well in your first month?
- What has been unclear or challenging?
- Do you feel you have the tools and access you need?
- What could we improve in the onboarding process?
- Are there any blockers or gaps preventing you from being fully productive?

**Actions**:
- [ ] Document feedback and update this checklist for future onboarding
- [ ] Address any outstanding access or knowledge gaps
- [ ] Confirm new team member feels confident in their role

---

## Using This Checklist

- Create a GitHub issue using this template for each new team member
- Assign the issue to the Project Manager or Scrum Master
- Tag relevant teammates who will support onboarding (buddy, tech leads, etc.)
- Use issue comments to track progress and share resources
- Close the issue after the 30-day check-in is complete
